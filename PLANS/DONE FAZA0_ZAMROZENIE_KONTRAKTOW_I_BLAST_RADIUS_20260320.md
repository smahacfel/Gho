# Faza 0 — Zamrożenie kontraktów i blast radius

**Data:** 2026-03-20  
**Baseline:** `main@567bc6005b5907b116987339a9a82289759ceae9`  
**Tryb:** inventory-only, zero zmian logiki produkcyjnej  
**Cel:** ustalić i zamrozić obecną prawdę operacyjną przed Fazą 1

## Metoda i reguły włączenia

- Źródłem prawdy jest wyłącznie kod z `main@567bc6`, nie wcześniejsze plany ani pamięć repo.
- Matrix writerów curve/history i blast radius obejmuje aktywny kod produkcyjny z workspace'owych ścieżek runtime: `src/`, `ghost-launcher/src/`, `ghost-core/src/`, `ghost-brain/src/`, `off-chain/components/seer/src/`.
- Alias-only mutacje `ShadowLedger`, które nie zapisują curve state ani canonical/live snapshot history, są spisywane osobno poza writer matrixem.
- Z prod matrixa wykluczone są `tests/`, `examples/`, `benches/` oraz inline callsite'y siedzące w blokach testowych `#[test]` / `#[tokio::test]`.
- Helper API bez potwierdzonego non-test prod-callera w tym baseline nie wchodzi do prod writer matrixa.
- Dla każdej ścieżki zapisano stan **dzisiejszy**, nawet jeśli jest architektonicznie błędny lub niekompletny.

### Wyłączenia z prod writer matrixa

- `push_snapshot()` i `push_snapshot_with_source()` pozostają **poza** canonical writer listą. Search po `src/` nie pokazał non-test prod-callera poza samym `ledger.rs`.
- Callsite'y `commit_history()` znalezione w `ghost-brain/src/oracle/engine.rs`, `ghost-brain/src/oracle/snapshot_engine.rs` i `ghost-brain/src/signals/market_signals.rs` w tym baseline siedzą w blokach testowych i zostały wyłączone.
- `ghost-core/src/shadow_ledger/gatekeeper.rs` istnieje jako osobny moduł, ale aktualny launcher runtime używa `ghost-launcher/src/components/gatekeeper.rs` + `GatekeeperCommitLoop`.

## Frozen Public Contracts for Later Phases

- Nie zmieniać pól ani semantyki `GeyserEvent::{Transaction, AccountUpdate, EntryAnchor}` bez aktualizacji wszystkich producerów i `Seer::process_event()`.
- Nie zmieniać pól ani polityk backpressure `SeerEvent` bez aktualizacji bridge'a `ghost-launcher/src/components/seer.rs`.
- Nie zmieniać payloadu `PoolTransaction`, zwłaszcza `event_ordinal`, `arrival_ts_ms`, `curve_data_known`, `curve_finality`, bez jednoczesnego audytu `SnapshotListener`, `OracleRuntime`, `Gatekeeper`.
- Nie poszerzać ani nie zwężać `GhostEvent::AccountUpdate` "po cichu". Obecne zawężenie względem IPC jest częścią zamrożonego kontraktu i musi być jawnie przeprojektowane dopiero w kolejnych fazach.
- Nie zmieniać semantyki `InitPoolEvent.timestamp_ms` ani `slot` bez jednoczesnego audytu `SnapshotEngine::handle_initialize_pool_event()`.
- Nie zmieniać znaczenia statusów `commit_history()` (`Persisted`, `NoOpExistingHistory`, `RejectedNoWrite`) w Fazie 0/1 bez aktualizacji callerów live i replay.
- Nie zmieniać guardów `append_live()` (`mint committed`, `slot != 0`, monotonic `tx_count`, unique `tx_key`, non-stale `timestamp_ms`) bez audytu live flush i startup replay flush.
- Nie repurpose'ować `WalRecord.ts_ms`; to pole jest dziś aktywnie używane jako watermark replay, mimo że nie spełnia docelowego kontraktu orderingowego.
- Nie dodawać nowego writera `ShadowLedger` bez dopisania nowego wiersza do tego matrixa i nazwania jego roli/P-rangi przed implementacją.
- Nie dodawać nowej alias-only mutacji `ShadowLedger` bez dopisania jej do osobnej sekcji alias inventory i jawnego rozstrzygnięcia, czy wpływa na precedence/recovery reasoning.
