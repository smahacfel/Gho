1. Executive Summary

Docelowa architektura Ghost Decision Stack V3 powinna być ewolucją obecnego pipeline’u, nie pełnym rewrite. Rdzeń V3 to risk-first staged decision funnel oparty wyłącznie o MaterializedFeatureSet jako kanoniczny snapshot decyzyjny. V3 nie ma zastępować obecnego Gatekeepera V2/V2.5 w P0. Ma działać najpierw jako shadow/evidence plane, który równolegle do obecnego verdictu wylicza własny risk verdict, opportunity verdict, evidence status, confidence breakdown i shadow verdict. Obecny kontrakt repo już wskazuje MaterializedFeatureSet oraz PoolObservationSession::materialize_features() jako główną granicę prawdy decyzyjnej, a AGENTS.md wyraźnie ostrzega przed legacy revival, bypassowaniem SSOT, generic rejectami i mieszaniem shadow/live.

V3 nie powinien być full rewrite, bo repo ma już dobre fundamenty: AccountStateCore, TxIntelligenceEngine, CheckpointEngine, GatekeeperBuffer, MaterializedFeatureSet, PDD, TAS, DOW, APS, typed reason codes i DecisionLogger schema v19. To są elementy, które należy uporządkować i rozszerzyć, nie wyrzucić. MaterializedFeatureSet już zawiera account features, tx-intel features, checkpoint features, curve readiness, sybil resistance, alpha fingerprint i tx_segment_sequence, czyli większość danych potrzebnych do V3.

V3 nie powinien być magicznym ML, bo obecnie najważniejszą potrzebą nie jest model statystyczny, tylko poprawna semantyka decyzji: co jest hard risk, co jest positive opportunity, co jest tylko degraded evidence, a co jest nieegzekwowalne. ML bez pełnych labeli shadow/replay/live-reconciliation zamieniłby jawny problem decyzyjny w czarną skrzynkę. Możliwa przyszła statystyczna kalibracja confidence jest P2/P3, nie P0.

V3 nie może być shortcutem score > threshold -> BUY. Taki model mieszałby risk rejection z opportunity selection i pozwalałby wysokiemu momentum przykryć PDD, sybil, curve, timestamp lub execution uncertainty. V3 ma rozdzielać risk_penalty, opportunity_score, evidence_status i confidence_caps.

V3 powinien działać najpierw jako shadow/evidence plane, bo current active behavior nie może zostać zmienione bez replay/shadow-burnin proof. Current rollout config deklaruje shadow mode (entry_mode = "shadow_only", execution_mode = "shadow"), a V2.5 config ma shadow_enabled = true i live_execution_enabled = false, więc naturalnym pierwszym krokiem jest dodatkowy, nierozstrzygający evaluator.

Najważniejsze przewagi V3 nad obecnym V2/V2.5: V3 formalizuje status dowodów, oddziela risk od opportunity, nie traktuje missing data jako clean pass, ma explicit confidence caps, rozdziela Early/Normal/Extended jako różne tryby decyzyjne, dodaje organic broadening i manipulation contradiction jako brakujące grupy interpretacyjne oraz dostarcza porównywalną telemetrykę current-vs-V3 bez zmiany active policy.

3. Design Doctrine for V3

Risk-first. W kodzie oznacza to, że V3 najpierw buduje RiskVerdict, a dopiero potem OpportunityVerdict. BUY nie może powstać, jeśli risk verdict jest hard_reject albo critical_unavailable. W danych oznacza to oddzielne pola risk status, risk reason i risk penalty. W logach oznacza to v3_shadow_risk_status, v3_shadow_risk_primary_reason, v3_shadow_risk_secondary_reasons.

Evidence-aware. Każda grupa dowodów ma status: clean, degraded, unavailable, insufficient_sample, stale, fallback, shadow_only, not_configured. W kodzie actionability nie jest globalną metodą EvidenceStatus; jest liczona per feature group i stage w V3 evaluatorze. W decyzji missing alpha, missing curve lub degraded sybil nie są clean pass. W logach każda decyzja musi mieć evidence status per module.

Staged decision funnel. V3 ma warstwy L0–L8. Każda warstwa ma jasno określone wyjścia: hard reject, pending, timeout, confidence cap albo positive evidence. W kodzie to powinno być evaluate_l1_identity, evaluate_l2_evidence, evaluate_l3_risk, evaluate_l5_opportunity, compute_l6_confidence, final_arbiter.

SSOT-first. V3 evaluator nie czyta raw tx ani mutable session state. Wszystko, co wymaga raw tx/ring buffer, musi zostać policzone w PoolObservationSession::materialize_features(). gatekeeper_v3.rs konsumuje tylko MaterializedFeatureSet i config. To zachowuje obecną granicę materialize_features() jako SSOT.

Deterministic. Ten sam MaterializedFeatureSet, ten sam config i ten sam policy version muszą dać ten sam V3 shadow verdict. W kodzie nie wolno odwoływać się do wall-clock, RPC ani runtime locks w evaluatorze.

Replayable. V3 output musi być możliwy do odtwórzenia z JSONL/replay: snapshot hash, config hash, decision plane, reason codes, evidence status, risk/opportunity/confidence breakdown.

Auditable. Każdy V3 BUY, REJECT, TIMEOUT, PENDING ma typed reason code, secondary reasons i stage. Generic reject jest niewystarczający.

Confidence-capped. Confidence nie może rosnąć przez brak danych. Degraded/unavailable evidence nakłada cap. W kodzie confidence model zwraca raw, after_risk, after_stage, cap, cap_reasons, final.

Shadow-first. P0 nie zmienia active GatekeeperBuffer::evaluate_from_features() ani active verdict behavior. V3 jest równoległą diagnozą, zapisywaną do logs/reports. Obecny repo contract wspiera shadow/live separation i DecisionLogger plane routing.

Low-latency. V3 ma być pure computation na materialized snapshot. Żadnego RPC, żadnego live sender, żadnego blokującego I/O, żadnego rozszerzania lock scope.

Execution-aware but not execution-coupled. V3 klasyfikuje pre-execution feasibility jako osobny status. W P0 execution evidence w pure evaluatorze to unavailable/execution_not_run. Shadow simulation outcome należy do logger/lifecycle plane, nie do pre-execution evaluatora. To respektuje zasadę: shadow simulation nie jest live inclusion. 

4. V3 Layered Architecture

L0 — Materialized evidence snapshot

Wejścia: obecny runtime/component state z PoolObservationSession, AccountStateCore, TxIntelligence, Checkpoints, GatekeeperBuffer, alpha/fingerprint, CPV/FSC.

Wyjścia: MaterializedFeatureSet rozszerzony addytywnie o V3 evidence fields.

Właściciel: PoolObservationSession::materialize_features().

Może emitować REJECT: nie. To warstwa materializacji, nie policy.

Może emitować PENDING/TIMEOUT: nie bezpośrednio; materializuje evidence status, który później prowadzi do pending/timeout.

Obniża confidence: pośrednio przez evidence status.

Telemetry: feature snapshot hash, materialization version, evidence status per group, fallback source.

Zasilające moduły: AccountStateCore, TxIntelligenceEngine, CheckpointEngine, GatekeeperBuffer, CPV/FSC, alpha fingerprint.

L1 — Identity / protocol / routing eligibility

Wejścia: session_metadata, pool/base mint/bonding curve identity, AccountStateCore phase, curve complete/migrated state, runtime identity registry outputs.

Wyjścia: IdentityEligibilityVerdict.

Właściciel: V3 policy, ale bazuje tylko na fields w MaterializedFeatureSet.

REJECT: invalid identity, unsupported protocol state, migrated/complete when execution path cannot handle it, duplicate terminal decision invariant.

PENDING: identity not yet resolvable before deadline, if current runtime supports waiting.

TIMEOUT: identity unresolved at deadline.

Confidence: identity degraded blocks BUY rather than merely reducing confidence.

Reason codes/telemetry: REJECT_INVALID_IDENTITY, REJECT_UNSUPPORTED_PROTOCOL_STATE, PENDING_V3_WAIT_EVIDENCE, TIMEOUT_V3_DEGRADED_EVIDENCE.

Repo inputs: SessionMetadata, AccountStateCore state phase, pool identity from runtime.

L2 — Data quality / evidence availability

Wejścia: MaterializedEvidenceStatus, curve readiness, sybil degraded reasons, optional alpha fields, tx counts, segment sequence, AccountStateCore update count.

Wyjścia: per-feature-group actionability, critical unavailable reasons, confidence caps.

Właściciel: V3 policy computes per-feature-group actionability; materialization owns raw statuses.

REJECT: only if data quality invalidates identity/protocol or proves corrupted evidence. Most unavailable evidence does not reject.

PENDING: critical evidence missing while time remains.

TIMEOUT: critical evidence missing at deadline.

Confidence: degraded/fallback/stale impose caps.

Reason/telemetry: PENDING_V3_WAIT_EVIDENCE, PENDING_V3_WAIT_SAMPLE, TIMEOUT_V3_DEGRADED_EVIDENCE, cap reasons, evidence status map.

Repo inputs: CurveReadinessFeatures, SybilResistanceFeatures.degraded_reasons, AlphaFingerprintFeatures, TxSegmentSequence.

L3 — Hard risk rejection

Wejścia: PDD diagnostics/materialized PDD-like fields, risk flags, tx-intel concentration/timing/failed/dust, sybil/CPV/FSC, curve sanity, dev/IWIM status when available.

Wyjścia: RiskVerdict.

Właściciel: V3 policy.

REJECT: yes, when hard risk is reliable/actionable.

PENDING: if hard risk cannot be evaluated because critical evidence is unavailable and time remains.

TIMEOUT: if hard risk cannot be evaluated by deadline and BUY cannot be justified.

Confidence: soft risk and degraded risk evidence reduce confidence.

Reason/telemetry: precise PDD reasons when matching existing codes; V3-specific manipulation contradiction only when no existing precise reason fits. Existing PDD reason codes already cover entry drift, spike, ramping, whale, reserve, flash crash.

Repo inputs: risk_flags, tx_intel_features, PDD/TAS/segment sequence, CPV/FSC fields, curve readiness.

L4 — Window and sample sufficiency

Wejścia: elapsed stage, tx_count, buy_count, unique_signers, segment sequence sample status, alpha sample, sybil sample.

Wyjścia: DecisionStage, SampleSufficiencyVerdict.

Właściciel: V3 policy; DOW config provides thresholds.

REJECT: only if sufficient sample proves low opportunity or hard risk.

PENDING: before sample minimum while time remains.

TIMEOUT: insufficient sample at deadline.

Confidence: insufficient but not critical can cap confidence.

Reason/telemetry: PENDING_V3_WAIT_SAMPLE, TIMEOUT_V3_DEGRADED_EVIDENCE, stage, sample observed/required.

Repo inputs: DOW config, tx-intel counts, TAS segment counts.

L5 — Opportunity selection

Wejścia: TAS, organic broadening, volume quality, alpha fingerprint, prosperity fit, curve health, buy pressure quality, signer diversity.

Wyjścia: OpportunityVerdict.

Właściciel: V3 policy consumes materialized features; organic/manipulation groups are materialized by session.

REJECT: low opportunity only when sample and evidence are sufficient.

PENDING: opportunity cannot be evaluated yet and time remains.

TIMEOUT: opportunity cannot be evaluated by deadline.

Confidence: opportunity components build positive score; missing components do not add score.

Reason/telemetry: BUY_V3_*, REJECT_V3_LOW_ORGANIC_BROADENING, REJECT_V3_LOW_OPPORTUNITY, component scores.

Repo inputs: TAS, AlphaFingerprintFeatures, prosperity diagnostics in current policy concepts, tx-intel metrics, curve fields.

L6 — Confidence model and uncertainty caps

Wejścia: risk verdict, opportunity verdict, evidence statuses, stage, execution status.

Wyjścia: ConfidenceBreakdown.

Właściciel: V3 policy.

REJECT: no direct reject; confidence can cause low opportunity reject at final arbiter.

PENDING: if confidence cannot be resolved due to waitable missing evidence.

TIMEOUT: unresolved confidence at deadline.

Confidence: this is the confidence layer.

Reason/telemetry: final confidence, cap reasons, raw score, risk penalty, stage multiplier.

L7 — Execution feasibility classification

Wejścia P0: none from execution; default execution = unavailable/execution_not_run.

Wejścia P1/P2: shadow simulation status, decision-to-sim latency, quote age, curve age, sim error class, compute units, blockhash/fee state.

Wyjścia: execution feasibility status and multiplier.

Właściciel P0: materialization defaults unavailable; logger/lifecycle owns actual shadow simulation outcome.

REJECT: only P1/P2 if deterministic execution infeasible is known before BUY. P0: no active reject.

PENDING/TIMEOUT: P0 can emit shadow pending/timeout if V3 model requires execution evidence for a hypothetical BUY, but should not affect current decision.

Confidence: execution unavailable caps confidence in V3 shadow only.

Reason/telemetry: execution evidence status, TIMEOUT_V3_UNRESOLVED_CONFIDENCE or future REJECT_EXECUTION_INFEASIBLE.

Repo boundary: do not touch DirectBuyBuilder/live sender/IWIM. Execution truth boundary is separate. 

L8 — Final arbiter

Wejścia: L1–L7 outputs.

Wyjścia: V3ShadowDecision { BUY | REJECT | TIMEOUT | PENDING }.

Właściciel: gatekeeper_v3.rs.

REJECT: hard risk, manipulation contradiction, low organic broadening, low opportunity with sufficient sample.

PENDING: waitable missing evidence/sample/window.

TIMEOUT: unresolved/degraded/insufficient by deadline.

BUY: clean risk-passed, opportunity-positive, confidence high enough, execution feasible or P0 pre-execution unknown with shadow-only caveat.

Telemetry: all fields required for replay and report.

6. EvidenceStatus Model

V3 should use evidence status as a typed description of proof quality, not as a global policy shortcut. Actionability is computed inside V3 per feature group, per stage.

clean means the evidence is present, timely, source-valid and adequate for its feature group. It can support hard reject if the feature group is a risk gate. It can support positive opportunity evidence. It does not impose confidence cap. It should be logged as {status: "clean"}.

degraded means the evidence exists but has a known weakness: partial parser confidence, provisional finality, partial CPV/FSC coverage, missing alpha subfield, fallback timestamp, etc. It can support hard reject only if the feature-specific policy allows degraded evidence for that risk. It should rarely support positive opportunity directly. It usually imposes confidence cap. It may lead to pending if the missing part can arrive within window. It may lead to timeout if unresolved by deadline. Log reason and source.

unavailable means no usable evidence for that group. It cannot support hard reject except identity/protocol corruption cases. It cannot support positive evidence. It imposes a cap or blocks BUY. If time remains, it can lead to pending. At deadline, it leads to timeout degraded/unresolved, not low-opportunity reject. Log exact unavailable reason.

insufficient_sample means evidence type exists but sample is below stage requirement. It cannot support hard reject unless the risk is independently confirmed. It cannot support positive opportunity. It imposes cap. It leads to pending if time remains and timeout if deadline. Log observed/required.

stale means evidence exists but age/finality/freshness is outside acceptable bound. It can support hard reject only if stale itself makes execution/protocol unsafe; otherwise it blocks BUY or caps confidence. Log age and freshness source.

fallback means evidence is derived from a fallback source, e.g. bootstrap candidate, GatekeeperBuffer curve dynamics, shadow ledger fallback. It may support some diagnostics, but not clean positive evidence. It usually caps confidence. It should never be hidden.

shadow_only means evidence is generated by shadow simulation/diagnostic plane. It cannot be treated as live inclusion. It may support validation and future promotion decisions. Log decision plane.

not_configured means a module is deliberately disabled. It should neither penalize nor reward unless policy requires the module for a stage. It must be logged separately from unavailable, because unavailable means expected-but-missing.

No global is_actionable() should exist. For example, degraded curve might be unacceptable for Early BUY but acceptable as cap in Normal; degraded alpha might be acceptable for Normal with cap; degraded CPV might block Early but not Normal if other sybil evidence is clean.

7. Hard Risk Gates

Identity/protocol hard gates

Condition: pool/mint/curve identity inconsistent, unsupported protocol state, migrated/complete when execution path expects bonding curve, duplicate terminal session invariant.

Required evidence: clean identity evidence; if identity is unavailable, pending/timeout, not reject unless contradiction is explicit.

Reason: P1/P2 should add REJECT_INVALID_IDENTITY, REJECT_UNSUPPORTED_PROTOCOL_STATE; P0 can use TIMEOUT_V3_DEGRADED_EVIDENCE or PENDING_V3_WAIT_EVIDENCE if no precise code exists.

Windows: all windows.

P0: shadow-only.

Promotion: after replay proves no false identity rejects.

Curve/market hard gates

Condition: curve not tradable, curve data clean and market/progress/reserve/price impact outside hard bounds, entry price already too drifted with reliable anchor.

Required evidence: curve clean or feature-specific degraded allowed. curve_data_known=false cannot be treated as clean hard gate.

Reason: existing HardFailMarketCap, HardFailTxPriceImpact, HardFailSellImpact, HardFailPriceChange can be reused; future exact V3 codes optional. 

Windows: all, with stricter Early.

P0: shadow-only.

Promotion: curve not tradable and confirmed price-impact hard gates are safest to promote first.

PDD hard gates

Condition: entry drift, spike, ramping, whale, reserve, flash crash. Existing PDD has typed PddHardFail values and reason mapping.

Required evidence: PDD-specific. Entry drift requires anchor quality; spike/ramping require sufficient sequence; flash crash requires actual price impact or price history. Path B flash crash is explicitly unavailable, not proxy. 

Reason: reuse RejectPddEntryDrift, RejectPddSpike, RejectPddRamping, RejectPddWhale, RejectPddReserve, RejectPddFlashCrash.

Windows: Early/Normal/Extended. Early requires stricter actionability.

P0: shadow-only.

Promotion: entry drift strong anchor, ramping, whale, reserve before spike/flash if validation supports.

Transaction/manipulation hard gates

Condition: extreme HHI, extreme top3, extreme bundling/same-ms, failed tx ratio, manipulation contradiction severe.

Required evidence: tx-intel clean and sample sufficient.

Reason: reuse existing hard fail reason codes for extreme HHI/top3/bundling/failed; add REJECT_V3_MANIPULATION_CONTRADICTION for composite contradiction. 

Windows: Normal/Extended; Early only if sample already high enough.

P0: shadow-only.

Promotion: only after ablation shows false BUY reduction without unacceptable opportunity loss.

Dev/IWIM hard gates

Condition: dev sold, IWIM rug/sybil/unknown strict.

Required evidence: dev wallet known or IWIM result exists. P0 V3 pre-execution should not run IWIM and should not change IWIM policy.

Reason: existing HardFailDevSold, RejectIwimVeto, RejectIwimLowConf, RejectIwimUnknownStrict.

Windows: all for dev sold; IWIM post-BUY only.

P0: V3 can log IWIM evidence as not_configured/unavailable in pure evaluator.

Promotion: no V3 promotion without separate IWIM ADR.

Sybil/infrastructure hard gates

Condition: CPV + FSC + fee topology / DBIA-like evidence confirm same-infra demand; extreme CPV; funding source concentration actionable and high.

Required evidence: CPV/FSC clean or sufficiently degraded but policy-approved; no hard reject from unavailable CPV/FSC.

Reason: existing RejectSybilCombo, RejectSybilInterference, RejectSybilSoftExcess; P1 can add exact REJECT_SYBIL_FSC_CONCENTRATION, REJECT_SYBIL_HIGH_CPV.

Windows: Normal/Extended; Early only if clean and high sample.

P0: shadow-only.

Promotion: later, only high-confidence combos.

Execution infeasibility hard gates

Condition: deterministic simulation/build failure, stale quote/curve beyond max age, deterministic slippage infeasible, blockhash/transaction invalidity. P0 pure evaluator does not have this evidence.

Required evidence: execution precheck/shadow lifecycle fields; not in P0 pure evaluator.

Reason: P1/P2 future REJECT_EXECUTION_INFEASIBLE; P0 uses ExecutionNotRun.

Windows: all BUY windows.

P0: classify unavailable only.

Promotion: requires execution-path review and live/shadow separation.

8. Opportunity Selection Model

TAS trajectory is positive when T0/T1/T2 show accelerating or stable healthy flow, HHI declining, volume consistent, intervals shortening without bundle pattern, buy ratio stable. TAS is noisy if sample per segment is low. It can be manipulated by synthetic segment bursts, so TAS must be cross-checked with organic broadening and manipulation contradiction. It is soft evidence, not standalone BUY. 

Organic broadening is positive when signer growth tracks tx growth, new signer ratio rises, HHI and top3 decline, repeat buyer pressure is low. It is a core V3 opportunity signal. It can become hard negative only if combined with high momentum contradiction.

Volume quality is positive when total volume is enough, volume distribution is not dominated by one signer/top3, volume CV is not pathological, single-tx sizes are not uniform ramping. Raw volume is noisy and redundant with buy pressure; quality matters more than quantity.

Alpha fingerprint is positive when fixed-size ratio and flipper presence are low, compute/static fee profile is not concentrated, early-slot dominance is not top3-only. It is soft evidence and a manipulation modifier. Missing alpha is not clean pass. Current alpha fields are optional in AlphaFingerprintFeatures, so V3 must track actionability. 

Prosperity fit is positive when candidate fits historically supported branches, but in V3 it should be logged as opportunity support, not hidden hard magic. APS can later suggest regime adjustments, but APS itself is shadow/offline and does not mutate live thresholds. 

Curve health is positive when curve readiness, finality/freshness, market cap, bonding progress and price impact are within configured bounds. Speculative/fallback curve should cap confidence rather than support high confidence.

Buy pressure quality is positive when buy pressure is broad, not uniform, not late-drifted, not dominated by repeated signers.

Signer diversity is both risk and opportunity. Extreme concentration is risk; healthy broadening is opportunity.

Early momentum quality is positive only if momentum is broadening, non-uniform, not drifted, not sybil-like and not execution-stale. High buy_count alone is not positive evidence.

10. Decision Windows: Early / Normal / Extended

Early

Early is not a loose fast-entry mode. Early BUY is allowed only when evidence is unusually clean and opportunity is strong.

Early BUY conditions:

no hard risk,

identity/curve/account evidence not critically unavailable,

sample above early minimum,

PDD actionable clean,

organic broadening positive,

no manipulation contradiction,

alpha either actionable positive or missing alpha cap still allows high confidence,

final confidence above early threshold,

execution status at least not known-infeasible; in P0 pure evaluator execution is unavailable and should be logged as such.


Early PENDING:

promising momentum but missing curve/account/PDD/alpha/sybil evidence,

insufficient sample,

stage before early min.


Early REJECT:

reliable hard risk,

severe manipulation contradiction with sufficient sample.


Early unavailable evidence that blocks BUY: curve/account missing, tx identity/timestamp untrusted, PDD sequence unavailable for a high-momentum candidate, severe sybil/funding degradation if configured as required for Early.

Normal

Normal is default decision path.

Normal BUY:

no hard risk,

sample sufficient,

critical evidence clean or degraded within caps,

opportunity_score strong/acceptable,

confidence above normal threshold,

no manipulation contradiction.


Normal REJECT:

hard risk,

or sufficient sample + low opportunity,

or low organic broadening with high pump-like activity.


Normal PENDING:

sample nearly sufficient,

curve/account delayed,

alpha/sybil/PDD not yet actionable and time remains,

confidence unresolved but not negative.


Extended

Extended is not a loosening mode. It is uncertainty resolution.

Extended BUY makes sense only when late evidence recovered a previous uncertainty and trajectory remains healthy: PDD clean, broadening improving, no new sell/whale/ramping, confidence high enough.

Extended REJECT:

newly confirmed hard risk,

sufficient sample + low opportunity,

deteriorating trajectory,

manipulation contradiction persists.


Extended TIMEOUT:

critical evidence unresolved by deadline,

insufficient sample by deadline,

confidence remains unresolved,

execution feasibility unknown when required.


Extended should reduce false BUY, not maximize late entries.

11. Confidence Model

V3 confidence must be deterministic, decomposed and logged.

Initial shadow hypothesis:

opportunity_score components:

TAS trajectory quality,

organic broadening,

volume quality,

alpha quality,

prosperity fit,

curve health,

buy pressure quality,

signer diversity.


risk_penalty components:

PDD soft risk,

manipulation contradiction,

sybil/infrastructure soft risk,

concentration soft risk,

timestamp/data-quality risk,

dev uncertainty,

curve fallback/finality risk.


stage_multiplier:

Early: slightly conservative despite high threshold,

Normal: neutral,

Extended: conservative unless late evidence recovered.


uncertainty_caps:

critical curve unavailable,

account bootstrap fallback,

timestamp fallback,

alpha unavailable,

sybil/CPV/FSC unavailable,

TAS/PDD sequence insufficient,

execution not run.


execution_multiplier:

P0 pure evaluator: unavailable/execution_not_run; do not treat as success.

P1/P2: simulation/build/quote/latency classification.


Final confidence:

final_confidence = min((opportunity_score - risk_penalty) * stage_multiplier * execution_multiplier, caps...)

Required telemetry:

raw opportunity score,

component scores,

risk penalty and components,

stage multiplier,

execution multiplier,

confidence cap,

cap reasons,

final confidence.


Thresholds must be config-driven in P1. P0 can use SHADOW_V3_* constants only if clearly shadow-only and logged as policy version.

12. Final Decision Tree

V3 shadow decision tree:

1. Materialized snapshot missing or invalid → PENDING_V3_WAIT_EVIDENCE if not deadline; TIMEOUT_V3_DEGRADED_EVIDENCE at deadline.


2. Identity/protocol contradiction → risk REJECT.


3. Critical evidence unavailable:

before deadline → PENDING,

at deadline → TIMEOUT.



4. Hard risk actionable → REJECT.


5. Sample insufficient:

before deadline → PENDING_V3_WAIT_SAMPLE,

at deadline → TIMEOUT_V3_DEGRADED_EVIDENCE.



6. Opportunity sufficient and risk clean and confidence above stage threshold → BUY.


7. Opportunity weak with sufficient clean sample → REJECT_V3_LOW_OPPORTUNITY.


8. Organic broadening weak with pump-like activity → REJECT_V3_LOW_ORGANIC_BROADENING or REJECT_V3_MANIPULATION_CONTRADICTION.


9. Confidence unresolved:

before deadline → PENDING_V3_WAIT_EVIDENCE,

at deadline → TIMEOUT_V3_UNRESOLVED_CONFIDENCE.




Semantics:

REJECT means negative proof or sufficient sample + low opportunity.

TIMEOUT means system lacks enough reliable proof by deadline.

PENDING means waitable state in V3 shadow only, not current active PendingCurve.

BUY means risk-passed, opportunity-positive, confidence high enough, and execution not known-infeasible. In P0 execution is not proven.

BUY precision should be measured only for V3 BUY candidates with enough outcome evidence. False BUY includes bad decision and execution-infeasible cases, but decision quality and execution quality must be separated.

Shadow simulation is not live inclusion. Submit is not confirmation. Unknown status is not success. These must be explicit in validation reports.
